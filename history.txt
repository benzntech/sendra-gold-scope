  ### 🔍 What was happening:

  When clicking on the links in the "Quick Shop" circles (like "Gold Chain", "Gold Bangle") or the footer links, the storefront was navigating to
  URLs like  http://13.140.136.162:3001/search/chain .
  However, the Next.js storefront returned a custom 404 page ("Well!, this page is as broken as our couch!").

  The reason this happened is because the Bagisto backend was recently populated with a 3-level deep category structure, creating very long and
  specific URL slugs for categories (e.g.,  traditional-silver-jewellery-and-articles-silver-women-jewellery-chain ). Because the frontend links
  were hardcoded to short URLs (like  chain  or  bangles ), the exact category match failed, triggering the 404 error page.

  ### 🛠️ How I fixed it:

  Instead of trying to hardcode extremely long, specific category slugs into every frontend component (which would break again if categories
  changed), I implemented a much smarter fallback in the routing logic:

  1. I modified the Next.js route for collections ( src/app/(public)/search/[collection]/page.tsx ).
  2. Now, if a user clicks a category link that doesn't perfectly match a valid backend slug (like  /search/chain ), the system will
  automatically redirect them to a keyword search for that exact term (i.e.,  /search?q=chain ).
  3. I tested the keyword search route, and it accurately queries the GraphQL backend and successfully renders all the relevant products (e.g.,
  displaying "18 KT GOLD ARYAN 18 PENDANT CHAIN" for the "chain" search).

  apply these code changes. Once the build finishes in a minute or two, all the previously broken links across your store will instantly start
  working by smartly falling back to product searches!
